Style Guideスタイルガイド

The following sections explain a few aspects of idiomatic Julia coding style. None of these rules are absolute; they are only suggestions to help familiarize you with the language and to help you choose among alternative designs.以下のセクションでは、慣用的なJuliaコーディングスタイルのいくつかの側面を説明します。これらのルールは絶対的なものではなく、言語に慣れるための提案であり、代替設計の中から選択する手助けをするためのものです。

Indentationインデント

Use 4 spaces per indentation level.インデントレベルごとに4つのスペースを使用してください。

Write functions, not just scripts関数を書き、単なるスクリプトにしないでください

Writing code as a series of steps at the top level is a quick way to get started solving a problem, but you should try to divide a program into functions as soon as possible. Functions are more reusable and testable, and clarify what steps are being done and what their inputs and outputs are. Furthermore, code inside functions tends to run much faster than top level code, due to how Julia's compiler works.最上位レベルで一連のステップとしてコードを書くことは、問題解決を始めるための迅速な方法ですが、できるだけ早くプログラムを関数に分割するように努めるべきです。関数は再利用可能でテスト可能であり、どのステップが実行されているのか、またその入力と出力が何であるかを明確にします。さらに、関数内のコードは、Juliaのコンパイラの動作により、最上位のコードよりもはるかに速く実行される傾向があります。

It is also worth emphasizing that functions should take arguments, instead of operating directly on global variables (aside from constants like pi).関数は引数を取るべきであり、グローバル変数（piのような定数を除く）に直接作用すべきではないことを強調する価値があります。

Avoid writing overly-specific types過度に特定の型を書くことを避ける

Code should be as generic as possible. Instead of writing:コードはできるだけ一般的であるべきです。次のように書くのではなく、

Complex{Float64}(x)

it's better to use available generic functions:利用可能な一般的な関数を使用する方が良いです：

complex(float(x))

The second version will convert x to an appropriate type, instead of always the same type.2番目のバージョンは、常に同じ型ではなく、xを適切な型に変換します。

This style point is especially relevant to function arguments. For example, don't declare an argument to be of type Int or Int32 if it really could be any integer, expressed with the abstract type Integer. In fact, in many cases you can omit the argument type altogether, unless it is needed to disambiguate from other method definitions, since a MethodError will be thrown anyway if a type is passed that does not support any of the requisite operations. (This is known as duck typing.)このスタイルのポイントは、特に関数の引数に関連しています。たとえば、引数が本当に任意の整数である場合、IntやInt32の型を宣言しないでください。実際、多くの場合、他のメソッド定義と区別するために必要でない限り、引数の型を完全に省略することができます。なぜなら、必要な操作をサポートしない型が渡された場合、MethodErrorが発生するからです。（これはダックタイピングとして知られています。）

For example, consider the following definitions of a function addone that returns one plus its argument:たとえば、引数に1を加える関数addoneの次の定義を考えてみてください：

addone(x::Int) = x + 1                 # works only for Int
addone(x::Integer) = x + oneunit(x)    # any integer type
addone(x::Number) = x + oneunit(x)     # any numeric type
addone(x) = x + oneunit(x)             # any type supporting + and oneunit

The last definition of addone handles any type supporting oneunit (which returns 1 in the same type as x, which avoids unwanted type promotion) and the + function with those arguments. The key thing to realize is that there is no performance penalty to defining only the general addone(x) = x + oneunit(x), because Julia will automatically compile specialized versions as needed. For example, the first time you call addone(12), Julia will automatically compile a specialized addone function for x::Int arguments, with the call to oneunit replaced by its inlined value 1. Therefore, the first three definitions of addone above are completely redundant with the fourth definition.最後のaddoneの定義は、oneunit（これはxと同じ型の1を返し、不要な型の昇格を避けます）をサポートする任意の型と、これらの引数での+関数を処理します。重要なことは、一般的なaddone(x) = x + oneunit(x)のみを定義することにパフォーマンスペナルティがないということです。なぜなら、Juliaは必要に応じて自動的に特化したバージョンをコンパイルするからです。たとえば、addone(12)を最初に呼び出すと、Juliaはx::Int引数用の特化したaddone関数を自動的にコンパイルし、oneunitへの呼び出しはそのインライン値1に置き換えられます。したがって、上記の最初の3つのaddoneの定義は、4番目の定義と完全に冗長です。

Handle excess argument diversity in the caller呼び出し元での過剰な引数の多様性を処理する

Instead of:代わりに：

function foo(x, y)
    x = Int(x); y = Int(y)
    ...
end
foo(x, y)

use:次のようにしてください：

function foo(x::Int, y::Int)
    ...
end
foo(Int(x), Int(y))

This is better style because foo does not really accept numbers of all types; it really needs Int s.これはより良いスタイルです。なぜなら、fooはすべてのタイプの数値を受け入れるわけではなく、実際にはIntが必要だからです。

One issue here is that if a function inherently requires integers, it might be better to force the caller to decide how non-integers should be converted (e.g. floor or ceiling). Another issue is that declaring more specific types leaves more "space" for future method definitions.ここでの一つの問題は、関数が本質的に整数を必要とする場合、呼び出し元に非整数をどのように変換するかを決定させる方が良いかもしれないということです（例：floorまたはceiling）。もう一つの問題は、より具体的なタイプを宣言することで、将来のメソッド定義のためにより多くの「スペース」が残ることです。

Append ! to names of functions that modify their arguments引数を変更する関数の名前に!を付ける

Instead of:代わりに：

function double(a::AbstractArray{<:Number})
    for i in eachindex(a)
        a[i] *= 2
    end
    return a
end

use:次のようにしてください：

function double!(a::AbstractArray{<:Number})
    for i in eachindex(a)
        a[i] *= 2
    end
    return a
end

Julia Base uses this convention throughout and contains examples of functions with both copying and modifying forms (e.g., sort and sort!), and others which are just modifying (e.g., push!, pop!, splice!). It is typical for such functions to also return the modified array for convenience.Julia Baseはこの規約を通じて使用されており、コピー形式と変更形式の両方の関数の例が含まれています（例：sortとsort!）、および単に変更するだけの関数（例：push!、pop!、splice!）。このような関数は、便利さのために変更された配列を返すことが一般的です。

Functions related to IO or making use of random number generators (RNG) are notable exceptions: Since these functions almost invariably must mutate the IO or RNG, functions ending with ! are used to signify a mutation other than mutating the IO or advancing the RNG state. For example, rand(x) mutates the RNG, whereas rand!(x) mutates both the RNG and x; similarly, read(io) mutates io, whereas read!(io, x) mutates both arguments.IOに関連する関数や乱数生成器（RNG）を利用する関数は顕著な例外です：これらの関数はほぼ必ずIOまたはRNGを変異させなければならないため、!で終わる関数は、IOを変異させることやRNGの状態を進めること以外の変異を示すために使用されます。例えば、rand(x)はRNGを変異させますが、rand!(x)はRNGとxの両方を変異させます。同様に、read(io)はioを変異させますが、read!(io, x)は両方の引数を変異させます。

Avoid strange type Unions奇妙なタイプUnionを避ける

Types such as Union{Function,AbstractString} are often a sign that some design could be cleaner.Union{Function,AbstractString}のようなタイプは、しばしば設計がよりクリーンであるべきであることを示す兆候です。

Avoid elaborate container types複雑なコンテナタイプを避ける

It is usually not much help to construct arrays like the following:以下のような配列を構築することは、通常あまり役に立ちません。

a = Vector{Union{Int,AbstractString,Tuple,Array}}(undef, n)

In this case Vector{Any}(undef, n) is better. It is also more helpful to the compiler to annotate specific uses (e.g. a[i]::Int) than to try to pack many alternatives into one type.この場合、Vector{Any}(undef, n) の方が良いです。また、特定の使用法（例：a[i]::Int）を注釈する方が、1つの型に多くの代替を詰め込もうとするよりもコンパイラにとっても有益です。

Prefer exported methods over direct field access直接フィールドアクセスよりもエクスポートされたメソッドを優先する

Idiomatic Julia code should generally treat a module's exported methods as the interface to its types. An object's fields are generally considered implementation details and user code should only access them directly if this is stated to be the API. This has several benefits:イディオマティックなJuliaコードは、一般的にモジュールのエクスポートされたメソッドをその型へのインターフェースとして扱うべきです。オブジェクトのフィールドは一般的に実装の詳細と見なされ、ユーザーコードはこれがAPIであると明記されている場合にのみ直接アクセスすべきです。これにはいくつかの利点があります：

  • Package developers are freer to change the implementation without breaking user code.パッケージ開発者は、ユーザーコードを壊すことなく実装を変更する自由があります。
  • Methods can be passed to higher-order constructs like map (e.g. map(imag, zs)) rather than [z.im for z in zs]).map のような高階構造にメソッドを渡すことができます（例：map(imag, zs)）が、[z.im for z in zs] のようにする必要はありません。
  • Methods can be defined on abstract types.メソッドは抽象型に対して定義できます。
  • Methods can describe a conceptual operation that can be shared across disparate types (e.g. real(z) works on Complex numbers or Quaternions).メソッドは、異なる型間で共有できる概念的な操作を説明できます（例：real(z) は複素数や四元数で動作します）。

Julia's dispatch system encourages this style because play(x::MyType) only defines the play method on that particular type, leaving other types to have their own implementation.Juliaのディスパッチシステムはこのスタイルを奨励します。なぜなら、play(x::MyType) はその特定の型に対してのみplayメソッドを定義し、他の型は独自の実装を持つことができるからです。

Similarly, non-exported functions are typically internal and subject to change, unless the documentations states otherwise. Names sometimes are given a _ prefix (or suffix) to further suggest that something is "internal" or an implementation-detail, but it is not a rule.同様に、エクスポートされていない関数は通常内部的なものであり、変更の対象となります。ドキュメントに別途記載がない限り。名前には、何かが「内部的」または実装の詳細であることを示唆するために_プレフィックス（またはサフィックス）が付けられることがありますが、これはルールではありません。

Counter-examples to this rule include NamedTuple, RegexMatch, StatStruct.このルールに対する反例には、NamedTuple、RegexMatch、StatStructが含まれます。

Use naming conventions consistent with Julia base/Julia base/と一貫した命名規則を使用する

  • modules and type names use capitalization and camel case: module SparseArrays, struct UnitRange.モジュールと型名は大文字とキャメルケースを使用します: module SparseArrays、struct UnitRange。
  • functions are lowercase (maximum, convert) and, when readable, with multiple words squashed together (isequal, haskey). When necessary, use underscores as word separators. Underscores are also used to indicate a combination of concepts (remotecall_fetch as a more efficient implementation of fetch(remotecall(...))) or as modifiers.関数は小文字で記述されます (maximum、convert)。可読性がある場合は、複数の単語を一緒に圧縮します (isequal、haskey)。必要に応じて、単語の区切りとしてアンダースコアを使用します。アンダースコアは、概念の組み合わせを示すためにも使用されます (remotecall_fetchは、fetch(remotecall(...))のより効率的な実装です) または修飾子として。
  • functions mutating at least one of their arguments end in !.引数のうち少なくとも1つを変更する関数は、!で終わります。
  • conciseness is valued, but avoid abbreviation (indexin rather than indxin) as it becomes difficult to remember whether and how particular words are abbreviated.簡潔さが重視されますが、略語は避けてください (indexinではなくindxin)。特定の単語がどのように略されているかを覚えるのが難しくなります。

If a function name requires multiple words, consider whether it might represent more than one concept and might be better split into pieces.関数名が複数の単語を必要とする場合、それが複数の概念を表している可能性があるかどうかを考慮し、分割する方が良いかもしれません。

Write functions with argument ordering similar to Julia BaseJulia Baseに似た引数の順序で関数を書く

As a general rule, the Base library uses the following order of arguments to functions, as applicable:一般的なルールとして、Baseライブラリは、適用可能な場合、関数の引数の次の順序を使用します:

  1. Function argument. Putting a function argument first permits the use of do blocks for passing multiline anonymous functions.関数引数。関数引数を最初に置くことで、doブロックを使用して複数行の無名関数を渡すことができます。

  2. I/O stream. Specifying the IO object first permits passing the function to functions such as sprint, e.g. sprint(show, x).I/Oストリーム。最初にIOオブジェクトを指定することで、sprintのような関数に関数を渡すことができます。例えば、sprint(show, x)のように。

  3. Input being mutated. For example, in fill!(x, v), x is the object being mutated and it appears before the value to be inserted into x.入力が変更される。例えば、fill!(x, v)では、xが変更されるオブジェクトであり、xに挿入される値の前に現れます。

  4. Type. Passing a type typically means that the output will have the given type. In parse(Int, "1"), the type comes before the string to parse. There are many such examples where the type appears first, but it's useful to note that in read(io, String), the IO argument appears before the type, which is in keeping with the order outlined here.型。型を渡すことは通常、出力が指定された型になることを意味します。parse(Int, "1")では、型が解析する文字列の前に来ます。このように型が最初に現れる例は多くありますが、read(io, String)のように、IO引数が型の前に現れることは、ここで示された順序に従っています。

  5. Input not being mutated. In fill!(x, v), v is not being mutated and it comes after x.入力が変更されない。fill!(x, v)では、vは変更されないものであり、xの後に来ます。

  6. Key. For associative collections, this is the key of the key-value pair(s). For other indexed collections, this is the index.キー。連想コレクションの場合、これはキー-値ペアのキーです。他のインデックス付きコレクションの場合、これはインデックスです。

  7. Value. For associative collections, this is the value of the key-value pair(s). In cases like fill!(x, v), this is v.値。連想コレクションの場合、これはキー-値ペアの値です。fill!(x, v)のような場合、これはvです。

  8. Everything else. Any other arguments.その他すべて。その他の引数。

  9. Varargs. This refers to arguments that can be listed indefinitely at the end of a function call. For example, in Matrix{T}(undef, dims), the dimensions can be given as a Tuple, e.g. Matrix{T}(undef, (1,2)), or as Varargs, e.g. Matrix{T}(undef, 1, 2).可変引数。これは、関数呼び出しの最後に無限にリストできる引数を指します。例えば、Matrix{T}(undef, dims)では、次元をTupleとして指定することができます。例えば、Matrix{T}(undef, (1,2))や、Varargとして、Matrix{T}(undef, 1, 2)のように指定できます。

  10. Keyword arguments. In Julia keyword arguments have to come last anyway in function definitions; they're listed here for the sake of completeness.キーワード引数。Juliaでは、キーワード引数は関数定義の最後に来る必要があります。完全性のためにここにリストされています。

The vast majority of functions will not take every kind of argument listed above; the numbers merely denote the precedence that should be used for any applicable arguments to a function.ほとんどの関数は、上記のすべての種類の引数を取るわけではありません。数字は、関数に適用可能な引数に対して使用すべき優先順位を示すものです。

There are of course a few exceptions. For example, in convert, the type should always come first. In setindex!, the value comes before the indices so that the indices can be provided as varargs.もちろん、いくつかの例外があります。例えば、convertでは、型は常に最初に来るべきです。setindex!では、値がインデックスの前に来るため、インデックスをvarargsとして提供できます。

When designing APIs, adhering to this general order as much as possible is likely to give users of your functions a more consistent experience.APIを設計する際には、この一般的な順序をできるだけ守ることが、関数のユーザーにより一貫した体験を提供する可能性が高いです。

Don't overuse try-catchtry-catchを過剰に使用しない

It is better to avoid errors than to rely on catching them.エラーを捕まえることに頼るよりも、エラーを避ける方が良いです。

Don't parenthesize conditions条件を括弧で囲まない

Julia doesn't require parens around conditions in if and while. Write:Juliaでは、ifやwhileの条件を括弧で囲む必要はありません。次のように書きます：

if a == b

instead of:次のように書かないでください：

if (a == b)

Don't overuse ...過剰に...を使用しない

Splicing function arguments can be addictive. Instead of [a..., b...], use simply [a; b], which already concatenates arrays. collect(a) is better than [a...], but since a is already iterable it is often even better to leave it alone, and not convert it to an array.関数の引数をスライスするのは中毒性があります。[a..., b...]の代わりに、単に[a; b]を使用してください。これはすでに配列を連結します。collect(a)は[a...]よりも良いですが、aはすでに反復可能なので、配列に変換せずにそのままにしておく方がさらに良いことが多いです。

Ensure constructors return an instance of their own typeコンストラクタが自分の型のインスタンスを返すことを確認する

When a method T(x) is called on a type T, it is generally expected to return a value of type T. Defining a constructor that returns an unexpected type can lead to confusing and unpredictable behavior:型 T に対してメソッド T(x) が呼び出されると、一般的には型 T の値を返すことが期待されます。予期しない型を返す コンストラクタ を定義すると、混乱を招き、予測不可能な動作を引き起こす可能性があります：

julia> struct Foo{T}
           x::T
       end

julia> Base.Float64(foo::Foo) = Foo(Float64(foo.x))  # Do not define methods like this

julia> Float64(Foo(3))  # Should return `Float64`
Foo{Float64}(3.0)

julia> Foo{Int}(x) = Foo{Float64}(x)  # Do not define methods like this

julia> Foo{Int}(3)  # Should return `Foo{Int}`
Foo{Float64}(3.0)

To maintain code clarity and ensure type consistency, always design constructors to return an instance of the type they are supposed to construct.コードの明確さを維持し、型の一貫性を確保するために、常にコンストラクタは構築すべき型のインスタンスを返すように設計してください。

Don't use unnecessary static parameters不要な静的パラメータを使用しない

A function signature:関数のシグネチャ：

foo(x::T) where {T<:Real} = ...

should be written as:次のように記述する必要があります：

foo(x::Real) = ...

instead, especially if T is not used in the function body. Even if T is used, it can be replaced with typeof(x) if convenient. There is no performance difference. Note that this is not a general caution against static parameters, just against uses where they are not needed.代わりに、特に T が関数本体で使用されていない場合は。たとえ T が使用されていても、便利であれば typeof(x) に置き換えることができます。パフォーマンスの違いはありません。これは静的パラメータに対する一般的な注意ではなく、必要ない場合の使用に対する注意です。

Note also that container types, specifically may need type parameters in function calls. See the FAQ Avoid fields with abstract containers for more information.また、コンテナ型は、特に関数呼び出しで型パラメータが必要な場合があります。詳細については、FAQ 抽象コンテナを持つフィールドを避ける を参照してください。

Avoid confusion about whether something is an instance or a type何かがインスタンスか型かについての混乱を避ける

Sets of definitions like the following are confusing:以下のような定義のセットは混乱を招きます：

foo(::Type{MyType}) = ...
foo(::MyType) = foo(MyType)

Decide whether the concept in question will be written as MyType or MyType(), and stick to it.問題の概念が MyType として書かれるのか MyType() として書かれるのかを決定し、それを守ってください。

The preferred style is to use instances by default, and only add methods involving Type{MyType} later if they become necessary to solve some problems.推奨されるスタイルは、デフォルトでインスタンスを使用し、問題を解決するために必要になった場合にのみType{MyType}に関するメソッドを追加することです。

If a type is effectively an enumeration, it should be defined as a single (ideally immutable struct or primitive) type, with the enumeration values being instances of it. Constructors and conversions can check whether values are valid. This design is preferred over making the enumeration an abstract type, with the "values" as subtypes.タイプが実質的に列挙型である場合、それは単一の（理想的には不変の構造体または原始型）タイプとして定義され、列挙値はそのインスタンスであるべきです。コンストラクタと変換は、値が有効かどうかをチェックできます。この設計は、列挙型を抽象型として作成し、「値」をサブタイプとするよりも好まれます。

Don't overuse macrosマクロを過剰に使用しない

Be aware of when a macro could really be a function instead.マクロが実際には関数であるべき時を意識してください。

Calling eval inside a macro is a particularly dangerous warning sign; it means the macro will only work when called at the top level. If such a macro is written as a function instead, it will naturally have access to the run-time values it needs.evalをマクロ内で呼び出すことは、特に危険な警告サインです。これは、マクロがトップレベルで呼び出されたときにのみ機能することを意味します。このようなマクロを関数として書くと、必要なランタイム値に自然にアクセスできるようになります。

Don't expose unsafe operations at the interface levelインターフェースレベルで安全でない操作を公開しない

If you have a type that uses a native pointer:ネイティブポインタを使用するタイプがある場合：

mutable struct NativeType
    p::Ptr{UInt8}
    ...
end

don't write definitions like the following:次のような定義を書かないでください：

getindex(x::NativeType, i) = unsafe_load(x.p, i)

The problem is that users of this type can write x[i] without realizing that the operation is unsafe, and then be susceptible to memory bugs.問題は、このタイプのユーザーがx[i]を安全でない操作であることに気づかずに書くことができ、その結果、メモリバグにさらされる可能性があることです。

Such a function should either check the operation to ensure it is safe, or have unsafe somewhere in its name to alert callers.そのような関数は、操作が安全であることを確認するか、呼び出し元に警告するためにunsafeを名前のどこかに含めるべきです。

Don't overload methods of base container types基本コンテナ型のメソッドをオーバーロードしない

It is possible to write definitions like the following:以下のような定義を書くことは可能です:

show(io::IO, v::Vector{MyType}) = ...

This would provide custom showing of vectors with a specific new element type. While tempting, this should be avoided. The trouble is that users will expect a well-known type like Vector() to behave in a certain way, and overly customizing its behavior can make it harder to work with.これは特定の新しい要素型を持つベクトルのカスタム表示を提供します。魅力的ではありますが、これは避けるべきです。問題は、ユーザーがVector()のようなよく知られた型が特定の方法で動作することを期待するため、その動作を過度にカスタマイズすると、扱いにくくなる可能性があることです。

Avoid type piracy型の盗用を避ける

"Type piracy" refers to the practice of extending or redefining methods in Base or other packages on types that you have not defined. In extreme cases, you can crash Julia (e.g. if your method extension or redefinition causes invalid input to be passed to a ccall). Type piracy can complicate reasoning about code, and may introduce incompatibilities that are hard to predict and diagnose."型の盗用"とは、あなたが定義していない型に対してBaseや他のパッケージでメソッドを拡張または再定義する行為を指します。極端な場合、あなたのメソッド拡張や再定義が無効な入力をccallに渡す原因となると、Juliaがクラッシュすることがあります。型の盗用はコードの推論を複雑にし、予測や診断が難しい互換性の問題を引き起こす可能性があります。

As an example, suppose you wanted to define multiplication on symbols in a module:例えば、モジュール内でシンボルの乗算を定義したいとします:

module A
import Base.*
*(x::Symbol, y::Symbol) = Symbol(x,y)
end

The problem is that now any other module that uses Base.* will also see this definition. Since Symbol is defined in Base and is used by other modules, this can change the behavior of unrelated code unexpectedly. There are several alternatives here, including using a different function name, or wrapping the Symbols in another type that you define.問題は、Base.*を使用する他のモジュールもこの定義を見ることになることです。SymbolはBaseで定義されており、他のモジュールでも使用されているため、無関係なコードの動作が予期せず変更される可能性があります。ここには、異なる関数名を使用するか、あなたが定義した別の型でSymbolをラップするなど、いくつかの代替手段があります。

Sometimes, coupled packages may engage in type piracy to separate features from definitions, especially when the packages were designed by collaborating authors, and when the definitions are reusable. For example, one package might provide some types useful for working with colors; another package could define methods for those types that enable conversions between color spaces. Another example might be a package that acts as a thin wrapper for some C code, which another package might then pirate to implement a higher-level, Julia-friendly API.時には、関連するパッケージが型の盗用に関与して、特徴を定義から分離することがあります。特に、パッケージが共同著者によって設計され、定義が再利用可能な場合です。例えば、あるパッケージが色を扱うために便利な型を提供し、別のパッケージがそれらの型に対して色空間間の変換を可能にするメソッドを定義することがあります。別の例としては、あるCコードの薄いラッパーとして機能するパッケージがあり、別のパッケージがそれを盗用して高レベルのJuliaフレンドリーなAPIを実装することがあります。

Be careful with type equality型の等価性に注意する

You generally want to use isa and <: for testing types, not ==. Checking types for exact equality typically only makes sense when comparing to a known concrete type (e.g. T == Float64), or if you really, really know what you're doing.一般的に、型をテストするためにはisaと<:を使用したいものであり、==は使用しない方が良いです。型の厳密な等価性をチェックすることは、通常、既知の具体的な型（例: T == Float64）と比較する場合や、あなたが本当に、本当に自分のやっていることを理解している場合にのみ意味があります。

Don't write a trivial anonymous function x->f(x) for a named function f名前付き関数 f のために、トリビアルな匿名関数 x->f(x) を書かないでください

Since higher-order functions are often called with anonymous functions, it is easy to conclude that this is desirable or even necessary. But any function can be passed directly, without being "wrapped" in an anonymous function. Instead of writing map(x->f(x), a), write map(f, a).高階関数はしばしば匿名関数と共に呼び出されるため、これが望ましい、あるいは必要であると結論づけるのは簡単です。しかし、任意の関数は匿名関数で「ラップ」されることなく直接渡すことができます。map(x->f(x), a)を書く代わりに、map(f, a)と書いてください。

Avoid using floats for numeric literals in generic code when possible可能な限り、汎用コードで数値リテラルに浮動小数点数を使用しないでください

If you write generic code which handles numbers, and which can be expected to run with many different numeric type arguments, try using literals of a numeric type that will affect the arguments as little as possible through promotion.数値を扱う汎用コードを書く場合、さまざまな数値型引数で実行されることが予想される場合は、昇格を通じて引数にできるだけ影響を与えない数値型のリテラルを使用するようにしてください。

For example,例えば、

julia> f(x) = 2.0 * x
f (generic function with 1 method)

julia> f(1//2)
1.0

julia> f(1/2)
1.0

julia> f(1)
2.0

whilewhile

julia> g(x) = 2 * x
g (generic function with 1 method)

julia> g(1//2)
1//1

julia> g(1/2)
1.0

julia> g(1)
2

As you can see, the second version, where we used an Int literal, preserved the type of the input argument, while the first didn't. This is because e.g. promote_type(Int, Float64) == Float64, and promotion happens with the multiplication. Similarly, Rational literals are less type disruptive than Float64 literals, but more disruptive than Ints:ご覧の通り、Intリテラルを使用した2番目のバージョンは、入力引数の型を保持しましたが、最初のバージョンはそうではありませんでした。これは、例えばpromote_type(Int, Float64) == Float64であり、昇格は乗算で発生するためです。同様に、Rationalリテラルは、Float64リテラルよりも型の破壊が少なく、Intよりも破壊的です：

julia> h(x) = 2//1 * x
h (generic function with 1 method)

julia> h(1//2)
1//1

julia> h(1/2)
1.0

julia> h(1)
2//1

Thus, use Int literals when possible, with Rational{Int} for literal non-integer numbers, in order to make it easier to use your code.したがって、可能な限りIntリテラルを使用し、リテラルの非整数値にはRational{Int}を使用して、コードの使用を容易にしてください。
