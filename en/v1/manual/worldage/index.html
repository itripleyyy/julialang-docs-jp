The World Age mechanismワールドエイジメカニズム

Note

World age is an advanced concept. For the vast majority of Julia users, the world age mechanism operates invisibly in the background. This documentation is intended for the few users who may encounter world-age related issues or error messages.ワールドエイジは高度な概念です。ほとんどのJuliaユーザーにとって、ワールドエイジメカニズムはバックグラウンドで目に見えない形で動作します。このドキュメントは、ワールドエイジに関連する問題やエラーメッセージに遭遇する可能性のある少数のユーザーを対象としています。

Julia 1.12

Prior to Julia 1.12, the world age mechanism did not apply to changes to the global binding table. The documentation in this chapter is specific to Julia 1.12+.Julia 1.12以前は、ワールドエイジメカニズムはグローバルバインディングテーブルの変更には適用されませんでした。この章のドキュメントはJulia 1.12以降に特有のものです。

Warning

This manual chapter uses internal functions to introspect world age and runtime data structures as an explanatory aid. In general, unless otherwise noted the world age mechanism is not a stable interface and should be interacted with in packages through stable APIs (e.g. invokelatest) only. In particular, do not assume that world ages are always integers or that they have a linear order.このマニュアルの章では、内部関数を使用して世界の年齢やランタイムデータ構造を調査し、説明の助けとしています。一般的に、特に記載がない限り、世界の年齢メカニズムは安定したインターフェースではなく、安定したAPI（例：invokelatest）を通じてパッケージ内でのみ相互作用するべきです。特に、世界の年齢が常に整数であるとか、線形順序を持つと仮定しないでください。

World age in general一般的な世界の年齢

The "world age counter" is a monotonically increasing counter that is incremented for every change to the global method table or the global binding table (e.g. through method definition, type definition, import/using declaration, creation of (typed) globals or definition of constants)."世界の年齢カウンター"は、グローバルメソッドテーブルやグローバルバインディングテーブルに対する変更ごとにインクリメントされる単調増加カウンターです（例：メソッド定義、型定義、import/using宣言、（型付き）グローバルの作成や定数の定義を通じて）。

The current value of the global world age counter can be retrieved using the (internal) function Base.get_world_counter.現在のグローバル世界の年齢カウンターの値は、（内部）関数Base.get_world_counterを使用して取得できます。

julia> Base.get_world_counter()
0x0000000000009632

julia> const x = 1

julia> Base.get_world_counter()
0x0000000000009633

In addition, each Task stores a local world age that determines which modifications to the global binding and method tables are currently visible to the running task. The world age of the running task will never exceed the global world age counter, but may run arbitrarily behind it. In general the term "current world age" refers to the local world age of the currently running task. The current world age may be retrieved using the (internal) function Base.tls_world_ageさらに、各Taskは、グローバルバインディングおよびメソッドテーブルに対する変更が現在実行中のタスクにどのように見えるかを決定するローカルな世界の年齢を保存します。実行中のタスクの世界の年齢は、グローバル世界の年齢カウンターを超えることはありませんが、それに対して任意に遅れることがあります。一般的に「現在の世界の年齢」という用語は、現在実行中のタスクのローカルな世界の年齢を指します。現在の世界の年齢は、（内部）関数Base.tls_world_ageを使用して取得できます。

julia> function f end
f (generic function with 0 methods)

julia> begin
           @show (Int(Base.get_world_counter()), Int(Base.tls_world_age()))
           Core.eval(@__MODULE__, :(f() = 1))
           @show (Int(Base.get_world_counter()), Int(Base.tls_world_age()))
           f()
       end
(Int(Base.get_world_counter()), Int(Base.tls_world_age())) = (38452, 38452)
(Int(Base.get_world_counter()), Int(Base.tls_world_age())) = (38453, 38452)
ERROR: MethodError: no method matching f()
The applicable method may be too new: running in current world age 38452, while global world is 38453.

Closest candidates are:
  f() (method too new to be called from this world context.)
   @ Main REPL[2]:3

Stacktrace:
 [1] top-level scope
   @ REPL[2]:5

julia> (f(), Int(Base.tls_world_age()))
(1, 38453)

Here the definition of the method f raised the global world counter, but the current world age did not change. As a result, the definition of f was not visible in the currently executing task and a MethodError resulted.ここでは、メソッドfの定義がグローバル世界カウンターを上昇させましたが、現在の世界の年齢は変わりませんでした。その結果、fの定義は現在実行中のタスクでは見えず、MethodErrorが発生しました。

Note

The method error printing provided additional information that f() is available in a newer world age. This information is added by the error display, not the task that threw the MethodError. The thrown MethodError is identical whether or not a matching definition of f() exists in a newer world age.メソッドエラーの印刷は、f()が新しい世界の年齢で利用可能であるという追加情報を提供しました。この情報は、MethodErrorをスローしたタスクではなく、エラー表示によって追加されます。スローされたMethodErrorは、f()の一致する定義が新しい世界の年齢に存在するかどうかにかかわらず同一です。

However, note that the definition of f() was subsequently available at the next REPL prompt, because the current task's world age had been raised. In general, certain syntactic constructs (in particular most definitions) will raise the current task's world age to the latest global world age, thus making all changes (both from the current task and any concurrently executing other tasks) visible. The following statements raise the current world age:ただし、f()の定義は次のREPLプロンプトで利用可能になったことに注意してください。これは、現在のタスクの世界の年齢が上昇したためです。一般的に、特定の構文構造（特にほとんどの定義）は、現在のタスクの世界の年齢を最新のグローバル世界の年齢に引き上げるため、すべての変更（現在のタスクからのものと、同時に実行されている他のタスクからのものの両方）が見えるようになります。次の文は現在の世界の年齢を上昇させます：

  1. An explicit invocation of Core.@latestworldCore.@latestworldの明示的な呼び出し
  2. The start of every top-level statementすべてのトップレベル文の開始
  3. The start of every REPL promptすべてのREPLプロンプトの開始
  4. Any type or struct definition任意の型または構造体の定義
  5. Any method definition任意のメソッド定義
  6. Any constant declaration任意の定数宣言
  7. Any global variable declaration (but not a global variable assignment)任意のグローバル変数宣言（ただし、グローバル変数の代入は除く）
  8. Any using, import, export or public statement任意のusing、import、exportまたはpublicステートメント
  9. Certain other macros like @eval (depends on the macro implementation)特定の他のマクロ、例えば@eval（マクロの実装に依存）

Note, however, that the current task's world age may only ever be permanently incremented at top level. As a general rule, using any of the above statements in non-top-level scope is a syntax error:ただし、現在のタスクのワールドエイジは、トップレベルでのみ永続的に増加することができます。一般的なルールとして、上記のステートメントを非トップレベルスコープで使用することは構文エラーです：

julia> f() = Core.@latestworld
ERROR: syntax: World age increment not at top level
Stacktrace:
 [1] top-level scope
   @ REPL[5]:1

When it isn't (for example for @eval), the world age side effect is ignored.それがそうでない場合（例えば@evalの場合）、ワールドエイジの副作用は無視されます。

As a result of these rules, Julia may assume that the world age does not change within the execution of an ordinary function.これらのルールの結果、Juliaは通常の関数の実行中にワールドエイジが変わらないと仮定することができます。

function my_function()
    before = Base.tls_world_age()
    # Any arbitrary code
    after = Base.tls_world_age()
    @assert before === after # always true
end

This is the key invariant that allows Julia to optimize based on the current state of its global data structures, while still having the well-defined ability to change these data structures.これは、Juliaがグローバルデータ構造の現在の状態に基づいて最適化を行うことを可能にする重要な不変条件であり、これらのデータ構造を変更する明確な能力を持っています。

Temporarily raising the world age using invokelatestinvokelatestを使用して世界の年齢を一時的に上げる

As described above, it is not possible to permanently raise the world age for the remainder of a Task's execution unless the task is executing top-level statements. However, it is possible to temporarily change the world age in a scoped manner using invokelatest:上記のように、タスクがトップレベルのステートメントを実行していない限り、Taskの実行の残りの間に世界の年齢を永続的に上げることはできません。しかし、invokelatestを使用してスコープ内で一時的に世界の年齢を変更することは可能です：

julia> function f end
f (generic function with 0 methods)

julia> begin
           Core.eval(@__MODULE__, :(f() = 1))
           invokelatest(f)
       end
1

invokelatest will temporarily raise the current task's world age to the latest global world age (at entry to invokelatest) and execute the provided function. Note that the world age will return to its prior value upon exit from invokelatest.invokelatestは、現在のタスクの世界の年齢を最新のグローバル世界の年齢（invokelatestへのエントリ時）に一時的に上げ、提供された関数を実行します。invokelatestから退出すると、世界の年齢は以前の値に戻ることに注意してください。

World age and const struct redefinitions世界の年齢と定数構造体の再定義

The semantics described above for method redefinition also apply to redefinition of constants:メソッドの再定義に関して上記で説明した意味論は、定数の再定義にも適用されます：

julia> const x = 1
1

julia> get_const() = x
get_const (generic function with 1 method)

julia> begin
           @show get_const()
           Core.eval(@__MODULE__, :(const x = 2))
           @show get_const()
           Core.@latestworld
           @show get_const()
       end
get_const() = 1
get_const() = 1
get_const() = 2
2

However, for the avoidance of doubt, they do not apply to ordinary assignment to global variables, which becomes visible immediately:ただし、誤解を避けるために、グローバル変数への通常の代入には適用されず、これは即座に可視化されます：

julia> global y = 1
1

julia> get_global() = y
get_global (generic function with 1 method)

julia> begin
           @show get_global()
           Core.eval(@__MODULE__, :(y = 2))
           @show get_global()
       end
get_global() = 1
get_global() = 2
2

One particular special case of constant reassignment is the redefinition of struct types:定数の再代入の特定の特別なケースは、構造体型の再定義です：

julia> struct MyStruct
           x::Int
       end

julia> const one_field = MyStruct(1)
MyStruct(1)

julia> struct MyStruct
           x::Int
           y::Float64
       end

julia> const two_field = MyStruct(1, 2.0)
MyStruct(1, 2.0)

julia> one_field
@world(MyStruct, 38452:38455)(1)

julia> two_field
MyStruct(1, 2.0)

Internally the two definitions of MyStruct are entirely separate types. However, after the new MyStruct type is defined, there is no longer any default binding for the original definition of MyStruct. To nevertheless facilitate access to these types, the special @world macro may be used to access the meaning of a name in a previous world. However, this facility is intended for introspection only and in particular note that world age numbers are not stable across precompilation and should in general be treated opaquely.内部的に、MyStructの2つの定義は完全に別々の型です。しかし、新しいMyStruct型が定義された後、元の定義のMyStructに対するデフォルトのバインディングはもはや存在しません。それでもこれらの型へのアクセスを容易にするために、特別な@worldマクロを使用して、以前の世界における名前の意味にアクセスできます。ただし、この機能は内部調査のためのものであり、特に世界の年齢番号はプリコンパイル間で安定しておらず、一般的には不透明に扱うべきであることに注意してください。

Binding partition introspectionバインディングパーティションの内部調査

In certain cases, it can be helpful to introspect the system's understanding of what a binding means in any particular world age. The default display printing of Core.Binding provides a helpful summary (e.g. on the MyStruct example from above):特定のケースでは、バインディングが特定のワールドエイジで何を意味するかについて、システムの理解を内省することが役立つ場合があります。Core.Bindingのデフォルト表示印刷は、役立つ要約を提供します（例：上記のMyStructの例）。

julia> convert(Core.Binding, GlobalRef(@__MODULE__, :MyStruct))
Binding Main.MyStruct
   38456:∞ - constant binding to MyStruct
   38452:38455 - constant binding to @world(MyStruct, 38452:38455)
   38451:38451 - backdated constant binding to @world(MyStruct, 38452:38455)
   0:38450 - backdated constant binding to @world(MyStruct, 38452:38455)

World age and using/importワールドエイジとusing/import

Bindings provided via using and import also operate via the world age mechanism. Binding resolution is a stateless function of the import and using definitions visible in the current world age. For example:usingおよびimportを介して提供されるバインディングも、ワールドエイジメカニズムを介して動作します。バインディング解決は、現在のワールドエイジで可視のimportおよびusing定義の無状態関数です。例えば：

julia> module M1; const x = 1; export x; end

julia> module M2; const x = 2; export x; end

julia> using .M1

julia> x
1

julia> using .M2

julia> x
ERROR: UndefVarError: `x` not defined in `Main`
Hint: It looks like two or more modules export different bindings with this name, resulting in ambiguity. Try explicitly importing it from a particular module, or qualifying the name with the module it should come from.

julia> convert(Core.Binding, GlobalRef(@__MODULE__, :x))
Binding Main.x
   38458:∞ - ambiguous binding - guard entry
   38457:38457 - implicit `using` resolved to constant 1

World age captureワールドエイジキャプチャ

Certain language features capture the current task's world age. Perhaps the most common of these is creation of new tasks. Newly created tasks will inherit the creating task's local world age at creation time and will retain said world age (unless explicitly raised) even if the originating tasks raises its world age:特定の言語機能は、現在のタスクのワールドエイジをキャプチャします。これらの中で最も一般的なのは、新しいタスクの作成です。新しく作成されたタスクは、作成時に作成タスクのローカルワールドエイジを継承し、元のタスクがワールドエイジを上げた場合でも、そのワールドエイジを保持します（明示的に上げられない限り）：

julia> const x = 1

julia> t = @task (wait(); println("Running now"); x);

julia> const x = 2

julia> schedule(t);
Running now

julia> x
2

julia> fetch(t)
1

In addition to tasks, opaque closures also capture their world age at creation. See Base.Experimental.@opaque.タスクに加えて、不透明なクロージャも作成時にそのワールドエイジをキャプチャします。詳細はBase.Experimental.@opaqueを参照してください。

Base.@world — Macro
@world(sym, world)

Resolve the binding sym in world world. See invoke_in_world for running arbitrary code in fixed worlds. world may be UnitRange, in which case the macro will error unless the binding is valid and has the same value across the entire world range.ワールドworldでバインディングsymを解決します。固定ワールドで任意のコードを実行するには、invoke_in_worldを参照してください。worldはUnitRangeである可能性があり、その場合、マクロはバインディングが有効であり、全ワールド範囲で同じ値を持つ場合を除いてエラーになります。

As a special case, the world ∞ always refers to the latest world, even if that world is newer than the world currently running.特別なケースとして、ワールド∞は常に最新のワールドを指し、そのワールドが現在実行中のワールドよりも新しい場合でも同様です。

The @world macro is primarily used in the printing of bindings that are no longer available in the current world.@worldマクロは、現在のワールドでもはや利用できないバインディングの印刷に主に使用されます。

Example例

julia> struct Foo; a::Int; end
Foo

julia> fold = Foo(1)

julia> Int(Base.get_world_counter())
26866

julia> struct Foo; a::Int; b::Int end
Foo

julia> fold

Base.invoke_in_world — Function
invoke_in_world(world, f, args...; kwargs...)

Call f(args...; kwargs...) in a fixed world age, world.f(args...; kwargs...) を固定されたワールドエイジ world で呼び出します。

This is useful for infrastructure running in the user's Julia session which is not part of the user's program. For example, things related to the REPL, editor support libraries, etc. In these cases it can be useful to prevent unwanted method invalidation and recompilation latency, and to prevent the user from breaking supporting infrastructure by mistake.これは、ユーザーのプログラムの一部ではないユーザーの Julia セッション内で実行されるインフラストラクチャにとって便利です。たとえば、REPL、エディタサポートライブラリなどに関連するものです。このような場合、不要なメソッドの無効化や再コンパイルの遅延を防ぎ、ユーザーが誤ってサポートインフラストラクチャを壊すのを防ぐことが有用です。

The global world age can be queried using Base.get_world_counter() and stored for later use within the lifetime of the current Julia session, or when serializing and reloading the system image.グローバルワールドエイジは、Base.get_world_counter() を使用して照会でき、現在の Julia セッションのライフタイム内で後で使用するために保存することができます。また、システムイメージをシリアライズおよび再ロードする際にも使用できます。

Technically, invoke_in_world will prevent any function called by f from being extended by the user during their Julia session. That is, generic function method tables seen by f (and any functions it calls) will be frozen as they existed at the given world age. In a sense, this is like the opposite of invokelatest.技術的には、invoke_in_world は、f によって呼び出される任意の関数がユーザーによってその Julia セッション中に拡張されるのを防ぎます。つまり、f（およびそれが呼び出す任意の関数）が見る汎用関数メソッドテーブルは、指定された world エイジで存在していたものとして凍結されます。ある意味では、これは invokelatest の逆のようなものです。

Note

It is not valid to store world ages obtained in precompilation for later use. This is because precompilation generates a "parallel universe" where the world age refers to system state unrelated to the main Julia session.プリコンパイルで取得したワールドエイジを後で使用するために保存することは無効です。これは、プリコンパイルがメインの Julia セッションとは無関係なシステム状態を参照する「平行宇宙」を生成するためです。

Base.Experimental.@opaque — Macro
@opaque ([type, ]args...) -> body

Marks a given closure as "opaque". Opaque closures capture the world age of their creation (as opposed to their invocation). This allows for more aggressive optimization of the capture list, but trades off against the ability to inline opaque closures at the call site, if their creation is not statically visible.指定されたクロージャを「不透明」としてマークします。不透明なクロージャは、その呼び出しではなく作成時のワールドエイジをキャプチャします。これにより、キャプチャリストのより積極的な最適化が可能になりますが、作成が静的に見えない場合、不透明なクロージャを呼び出し元でインライン化する能力とトレードオフになります。

An argument tuple type (type) may optionally be specified, to specify allowed argument types in a more flexible way. In particular, the argument type may be fixed length even if the function is variadic.引数タプル型 (type) をオプションで指定することができ、許可される引数型をより柔軟に指定できます。特に、関数が可変引数であっても、引数型は固定長である場合があります。
