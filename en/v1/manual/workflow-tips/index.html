Workflow Tipsワークフローヒント

Here are some tips for working with Julia efficiently.Juliaを効率的に操作するためのヒントをいくつか紹介します。

REPL-based workflowREPLベースのワークフロー

As already elaborated in The Julia REPL, Julia's REPL provides rich functionality that facilitates an efficient interactive workflow. Here are some tips that might further enhance your experience at the command line.The Julia REPLで詳しく説明されているように、JuliaのREPLは効率的なインタラクティブワークフローを促進する豊富な機能を提供します。コマンドラインでの体験をさらに向上させるためのヒントをいくつか紹介します。

A basic editor/REPL workflow基本的なエディタ/REPLワークフロー

The most basic Julia workflows involve using a text editor in conjunction with the julia command line.最も基本的なJuliaのワークフローは、テキストエディタとjuliaコマンドラインを組み合わせて使用することです。

Create a file, say Tmp.jl, and include within itファイルを作成します。例えばTmp.jlという名前にし、その中に含めます。

module Tmp

say_hello() = println("Hello!")

# Your other definitions here

end # module

using .Tmp

Then, in the same directory, start the Julia REPL (using the julia command). Run the new file as follows:次に、同じディレクトリでJulia REPLを開始します（juliaコマンドを使用）。新しいファイルを次のように実行します：

julia> include("Tmp.jl")

julia> Tmp.say_hello()
Hello!

Explore ideas in the REPL. Save good ideas in Tmp.jl. To reload the file after it has been changed, just include it again.REPLでアイデアを探求します。良いアイデアはTmp.jlに保存します。ファイルが変更された後に再読み込みするには、再度includeします。

The key in the above is that your code is encapsulated in a module. That allows you to edit struct definitions and remove methods, without restarting Julia.上記の重要な点は、あなたのコードがモジュールにカプセル化されていることです。これにより、struct定義を編集したり、メソッドを削除したりすることができ、Juliaを再起動する必要がありません。

(Explanation: structs cannot be edited after definition, nor can methods be deleted. But you can overwrite the definition of a module, which is what we do when we re-include("Tmp.jl")).(説明：structは定義後に編集できず、メソッドも削除できません。しかし、モジュールの定義を上書きすることはできます。これが、再度include("Tmp.jl")を実行する理由です。)

In addition, the encapsulation of code in a module protects it from being influenced by previous state in the REPL, protecting you from hard-to-detect errors.さらに、コードをモジュールにカプセル化することで、REPL内の以前の状態に影響されることから保護され、検出が難しいエラーから守られます。

Browser-based workflowブラウザベースのワークフロー

There are a few ways to interact with Julia in a browser:ブラウザでJuliaと対話する方法はいくつかあります：

  • Using Pluto notebooks through Pluto.jlPluto.jlを通じてPlutoノートブックを使用することです。
  • Using Jupyter notebooks through IJulia.jlJupyterノートブックを通じて IJulia.jl を使用する

Revise-based workflowsReviseベースのワークフロー

Whether you're at the REPL or in IJulia, you can typically improve your development experience with Revise. It is common to configure Revise to start whenever julia is started, as per the instructions in the Revise documentation. Once configured, Revise will track changes to files in any loaded modules, and to any files loaded in to the REPL with includet (but not with plain include); you can then edit the files and the changes take effect without restarting your julia session. A standard workflow is similar to the REPL-based workflow above, with the following modifications:REPLでもIJuliaでも、通常はReviseを使用することで開発体験を向上させることができます。juliaが起動するたびにReviseを開始するように設定するのが一般的で、これはReviseのドキュメントの指示に従います。設定が完了すると、Reviseは読み込まれたモジュール内のファイルの変更や、includetを使用してREPLに読み込まれたファイルの変更を追跡します（ただし、通常のincludeではありません）。その後、ファイルを編集すると、juliaセッションを再起動することなく変更が反映されます。標準的なワークフローは、上記のREPLベースのワークフローに似ていますが、以下の変更があります：

  1. Put your code in a module somewhere on your load path. There are several options for achieving this, of which two recommended choices are:コードをロードパス上のどこかのモジュールに配置します。これを達成するためのいくつかのオプションがあり、その中で推奨される2つの選択肢は次のとおりです：

    • For long-term projects, use PkgTemplates:長期プロジェクトの場合は、PkgTemplatesを使用します：

      using PkgTemplates
t = Template()
t("MyPkg")

      This will create a blank package, "MyPkg", in your .julia/dev directory. Note that PkgTemplates allows you to control many different options through its Template constructor.これにより、"MyPkg"という空のパッケージが、.julia/devディレクトリに作成されます。PkgTemplatesは、そのTemplateコンストラクタを通じて多くの異なるオプションを制御できることに注意してください。

      In step 2 below, edit MyPkg/src/MyPkg.jl to change the source code, and MyPkg/test/runtests.jl for the tests.以下のステップ2では、MyPkg/src/MyPkg.jlを編集してソースコードを変更し、MyPkg/test/runtests.jlをテスト用に編集します。

    • For "throw-away" projects, you can avoid any need for cleanup by doing your work in your temporary directory (e.g., /tmp)."使い捨て"プロジェクトの場合は、一時ディレクトリ（例：/tmp）で作業することで、クリーンアップの必要を回避できます。

      Navigate to your temporary directory and launch Julia, then do the following:一時ディレクトリに移動し、Juliaを起動して、次の操作を行います：

      pkg> generate MyPkg            # type ] to enter pkg mode
julia> push!(LOAD_PATH, pwd())   # hit backspace to exit pkg mode

      If you restart your Julia session you'll have to re-issue that command modifying LOAD_PATH.Juliaセッションを再起動すると、LOAD_PATHを変更するコマンドを再度発行する必要があります。

      In step 2 below, edit MyPkg/src/MyPkg.jl to change the source code, and create any test file of your choosing.以下のステップ2では、MyPkg/src/MyPkg.jlを編集してソースコードを変更し、任意のテストファイルを作成します。

  2. Develop your packageパッケージを開発する

    Before loading any code, make sure you're running Revise: say using Revise or follow its documentation on configuring it to run automatically.コードを読み込む前に、Reviseが実行されていることを確認してください：using Reviseと入力するか、自動的に実行するための設定に関するドキュメントを参照してください。

    Then navigate to the directory containing your test file (here assumed to be "runtests.jl") and do the following:次に、テストファイルを含むディレクトリに移動します（ここでは"runtests.jl"と仮定します）そして、以下を実行します：

    julia> using MyPkg

julia> include("runtests.jl")

    You can iteratively modify the code in MyPkg in your editor and re-run the tests with include("runtests.jl"). You generally should not need to restart your Julia session to see the changes take effect (subject to a few limitations).エディタでMyPkgのコードを反復的に修正し、include("runtests.jl")でテストを再実行できます。変更を反映させるためにJuliaセッションを再起動する必要は通常ありません（いくつかの制限を除いて）。
